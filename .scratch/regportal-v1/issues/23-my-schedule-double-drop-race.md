## Bug

`confirmAction()` in `ScheduleView` sets `pendingAction = null` synchronously before awaiting the drop. This means the confirmation dialog closes the instant the student clicks "Yes, Drop", but the section remains visible in the list (it isn't removed until `_applyResult` runs after the response). If the API is slow, the student sees the section still listed with its Drop button active and can click Drop again, firing a second concurrent request for the same section. The second request hits the backend for a section that may no longer be enrolled, returns an error, and `_applyResult` writes that error to `sectionErrors` — leaving a phantom error banner on the section after the first drop ultimately succeeds.

## Agent Brief

**Category:** bug
**Summary:** Double-drop race: pendingAction cleared before await allows concurrent drop requests on the same section

**Current behavior:**
`confirmAction()` nulls `pendingAction` before `await drop(courseKey)`. The dialog closes immediately. The section remains in `authStore.currentCourses` until `_applyResult` runs after the response, so the Drop button is re-enabled (it appears in `summaryList`) during the in-flight request. A second click fires a concurrent drop.

**Desired behavior:**
The section's Drop button should be disabled while its drop is in-flight, preventing a second request. The established pattern in this codebase for in-flight state is a loading `Set` of courseKeys (see `useRegisterNow`'s `registeringSections`). Apply the same pattern: add the sectionId to a `droppingSections` Set before awaiting and remove it in a `finally` block. The Drop button should be disabled (or hidden) for any section whose courseKey is in `droppingSections`.

**Key interfaces:**
- `ScheduleView` setup — add a `droppingSections` reactive `Set` (or `ref(new Set())`); expose it to the template
- `confirmAction()` — add the courseKey to `droppingSections` before the await; remove it in finally
- Drop button in the summary list and mobile list — disable when `droppingSections.has(entry.courseKey)`
- Waitlist Drop button — same guard

**Acceptance criteria:**
- [ ] Clicking "Yes, Drop" disables the Drop button for that section immediately
- [ ] The button remains disabled until the API response is received (success or error)
- [ ] On success, the section is removed from the list (button gone)
- [ ] On error, the button is re-enabled (or replaced by the error banner)
- [ ] A second click of Drop on the same section while in-flight is not possible
- [ ] Sections not being dropped are unaffected

**Out of scope:**
- Cancelling an in-flight drop request
- Global loading indicator
- Changes to the confirmation dialog UX
