## Bug

In `useScheduleRegistration._applyResult`, the expression `data.rows[0]?.errors` has a subtle optional-chaining gap: `data.rows[0]` is evaluated before `?.`, so if `data.rows` is `undefined` (unexpected API shape — maintenance page, empty body, or a response with no `rows` key), JavaScript throws `TypeError: Cannot read properties of undefined (reading '0')` before the optional chain has any effect.

All test fixtures include `rows: [...]` so the tests pass, but production is exposed to any off-contract response shape from the backend.

## Agent Brief

**Category:** bug
**Summary:** `data.rows[0]?.errors` throws TypeError when `rows` is absent from API response

**Current behavior:**
`_applyResult(data, ...)` reads `data.rows[0]?.errors ?? []`. The `?.` only guards the `.errors` property access; it does not protect against `data.rows` itself being `undefined`. If the backend returns a response body without a `rows` field, `data.rows[0]` throws a TypeError.

**Desired behavior:**
The expression should use safe subscript notation: `data.rows?.[0]?.errors ?? []`. This returns `[]` for any of: `data.rows` is undefined, `data.rows` is empty, `data.rows[0]` has no `errors` field.

**Key interfaces:**
- `_applyResult(data, sectionId, source)` in `useScheduleRegistration` — the single line `const errors = data.rows[0]?.errors ?? []` should become `data.rows?.[0]?.errors ?? []`

**Acceptance criteria:**
- [ ] `_applyResult` called with `{ rows: undefined }` does not throw — treats errors as `[]` and removes the section
- [ ] `_applyResult` called with `{}` (no rows key) does not throw
- [ ] `_applyResult` called with `{ rows: [] }` does not throw — treats errors as `[]`
- [ ] Normal happy-path behavior (rows with errors or no errors) is unchanged

**Out of scope:**
- Logging or reporting unexpected response shapes
- Changes to any other optional-chaining in the codebase
