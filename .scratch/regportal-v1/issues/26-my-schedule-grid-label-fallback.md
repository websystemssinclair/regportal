## Bug

The weekly grid block label in `ScheduleView` is built from `SubjectCode` and `CourseNo`: `` `${sec.SubjectCode?.trim()}-${sec.CourseNo?.trim()}` ``. If either field is `null` or `undefined` (a Colleague data gap or a special-topics section), the label renders as the literal string `"undefined-undefined"`, visible to the student in every grid block and mobile day-strip card for that section.

The summary list correctly uses `sec.LongName` for display, but `buildBlocks` (which feeds both the desktop grid and the mobile day-strip) has no fallback.

## Agent Brief

**Category:** bug
**Summary:** Grid block and mobile day-strip card label shows "undefined-undefined" when SubjectCode or CourseNo is absent

**Current behavior:**
`buildBlocks()` sets `label: \`${sec.SubjectCode?.trim()}-${sec.CourseNo?.trim()}\``. When `SubjectCode` or `CourseNo` is null/undefined, the template literal produces `"undefined-undefined"` (optional chaining on `.trim()` returns `undefined`, which stringifies as `"undefined"`).

**Desired behavior:**
If `SubjectCode` or `CourseNo` is absent, fall back to `sec.LongName` (the same field used by `summaryList`). If `LongName` is also absent, fall back to the raw `CourseKey`.

**Key interfaces:**
- `buildBlocks()` in `ScheduleView` — update the `label` computation
- The label must still be short enough to fit in a grid block cell; if falling back to `LongName`, consider truncating to ~20 characters

**Acceptance criteria:**
- [ ] A section with valid `SubjectCode` and `CourseNo` shows the existing `SUBJ-COURSENO` format (e.g. `ACC-1100`)
- [ ] A section where `SubjectCode` is null falls back to `LongName`
- [ ] A section where `CourseNo` is null falls back to `LongName`
- [ ] A section where both `SubjectCode` and `CourseNo` are null falls back to `LongName`
- [ ] A section where `LongName` is also null falls back to `CourseKey`
- [ ] The label never contains the literal string "undefined"

**Out of scope:**
- Changes to `summaryList` label rendering (already uses LongName correctly)
- Truncation logic beyond preventing "undefined-undefined"
