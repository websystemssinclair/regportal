## Bug (disputed — needs design decision)

`_applyResult` in `useScheduleRegistration` uses an if/else: if the target section is **not** in the API's error list, it is removed (success); if it **is** in the error list, all errors in the response are written to `sectionErrors`. This means that if the API returns a success for the target section **and** a collateral error for a different section (e.g. a prerequisite warning triggered by the drop), the else branch is never entered and the collateral error is silently discarded.

**Why this is needs-triage:** The current payload always sends exactly one section per request, so the backend returning an error for a section other than the one being dropped would be unusual behaviour. This may be impossible in practice given how the Colleague backend works, in which case the current if/else is correct and this issue is `wontfix`. Maintainer should decide:

1. **Wontfix** — the Colleague backend never returns collateral errors on a drop for other sections. The if/else is intentionally scoped to the targeted section only.
2. **Fix** — always write all errors from the response to `sectionErrors`, regardless of whether the target section succeeded, and separately determine success/removal by checking only the target section's presence in `failedKeys`.
